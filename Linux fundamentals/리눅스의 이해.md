# Section 1. 리눅스의 개요
## [리눅스의 구조와 주요 특징]

### [1] 리눅스의 특징 및 장단점
(1) 특징
- 오픈소스 운영체제 : 소스코드 및 관련자료들이 모두 공개되어 있음.
  
- 멀티유저 운영체제 : 여러 사용자가 동시에 같은 시스템에 접근 가능
  
- 멀티태스킹 운영체제 : 한 번에 여러개의 프로그램을 동시에 실행가능하다(가상 터미널 활용)
  
- 다중 스레드 운영체제 : 한 프로그램 내에서 병렬작업이 가능하다.
  
- 여러 종류의 파일시스템을 지원하는 운영체 : 리눅스 기본 파일 시스템 (ext2, ext3, ext4)외에도 DOS의 FAT16, 윈도우의 FAT32, NTFS, 네트워크 파일 시스템 SMB, CIFS, NFS등을 지원. 


(2) 장단점
- 유닉스와 높은 호환성

- PC용 운영체제보다 안정적 : 네트워크 사용을 전제로 설계되었기 때문에 항상 켜놓아도 안정적으로 운영된다.
  - 리눅스는 일반 윈도우 PC와 다르게 처음부터 문서작업 보다는 "연결"을 위해 만들어짐.
  - 사용자는 멀리있고, 네트워크를 통해 접속하는 상황을 당연하게 여김
  - 데몬 구조 : 리눅스에서 돌아가는 수많은 프로그램들은 네트워크 요청이 오기만을 기다리는 대기상태로 24시간 깨어 있을 수 있도로 설계되었다.
  - 윈도우는 한 명이 로그인하면 다른 사람은 사용하지 못하게 되지만 리눅스는 같은 리눅스 서버 안에서 여러명의 사용자가 각자의 작업을 할 수 있다.
 
- 하드웨어 기능을 효과적으로 사용 : 다른 운영체제보다 적은 양의 메모리를 필요한다.
  - RAM 최적화: 자주 쓰는 프로그램(nginx 등)은 RAM에, 백그라운드 또는 덜 쓰는 프로그램은 swap에 배치 -> 메모리 효율성 ↑

- 오픈 소스 운영체제 : 운영체제 뿐만 아니라, 여러 가지 유틸리티 프로그램과 응용 프로그램들을 사용자 편의에 맞춰 무료로 사용할 수 있다. 

-  공개 운영체제적 특성에서 비롯된 기술 지원의 어려움 : RHEL, SUSE등 몇개를 제외하고는 대부분은 예상치 못한 오류 발생시 개발자들의 기술 지원을 직접적으로 받기가 어려움.

- 한글 지원의 미흡 : 배포판마다 별도의 한글 지원 패키지를 설치한 후 사용해야함.

- 보안상 취약점의 노출 가능성 : 공개 운영체제이지만 꾸준한 기술 개발로 비교적 높은 보안성을 지원하며 많은 프로그래머들이 리눅스를 연구하고 있기 때문에, 보안 문제가 발생하였을 경우 신속하게 해결될 수 있다.
  
### [2] 운영체제의 구조와 기능
(1) 디렉토리
- 디렉토리 = 파일들의 저장소
- 리눅스 디렉토리는 최상위 디렉토리인 / 을 기준으로 하위 디렉토리들이 존재하는 "계층적 트리 구조로 구성되어 있다"
- 디렉토리 간에는 부모와 자식의 관계를 가지므로 상위 디렉토리와 하위 디렉토리는 부모 디렉토리와 자식디렉토리로 구분한다.

(2) 디렉토리별 저장 내용

- "/" : 최상위 디렉토리, 모든 디렉토리의 출발점
- 
- "/boot" : 부트 디렉토리, 부팅 시 커널 이미지와 부팅 정보 파일 저장
- 
- "/proc" : 시스템 정보 디렉토리, 커널 기능을 제어함. 현재 실행되고 있는 프로세스들, 실제로 사용되는 장치와 하드웨어 정보들을 저장한다.
  - 가상 파일 시스템
  - 시스템에서 운영되고 있는 다양한 프로세스들에 관한 내용과 프로그램에 대한 정보를 포함한다.
  - 여기서 볼 수 있는 건 메모리상에 저장되어 있다.
  - 사용자가 /proc나 하위파일에 접근할 때마다 커널에서 파일 내용을 동적으로 만들어낸다.
  - 각 프로세스는 고유의 식별자를 가지고 있으며, 이 식별자를 가진 디렉터리 밑에 정보를 저장한다.
 
- "/lib" : 공유 라이브러리 디렉토리, 커널 모듈 파일들과 프로그램 실행을 위한 라이브러리들을 저장한다. -> 실제로는 /usr/lib 의 심볼릭 링크이다. 
  - 동적 공유 라이브러리를 저장하고 있다.
  - 공유 라이브러리에는 많은 프로그램에서 공통으로 사용하는 함수들이 들어있어 디스크의 공간을 절약할 수 있으며, 프로그램마다 동일한 코딩을 할 필요가 없다. (/lib/libc.so -> C언어 표준 라이브러)
  - 라이브러리 공유 방법에는 "정적 라이브러리"와 "동적 라이브러리" 두 가지 방법이 있다.
  - 정적 라이브러리 : 컴파일 시 필요한 라이브러리 함수(서브루틴) 코드를 복사해서 실행 파일 안에 넣어버림.
  - 동적 라이브러리 : 실행 파일 내부에 라이브러리 함수 코드를 넣어두지 않고 실행시 필요할 떄 가져와서 사용하여 메모리의 효율성이 높다. 

- "/bin" : 기본적인 명령어가 저장된 디렉토리, root 사용자와 일반 사용자가 함꼐 사용할 수 있는 명령어들을 모은 명령어 디렉토리. (cp, ls, mv , cat 등등) -> 실제로는 /usr/bin경로의 심볼링 링크이다. 
  
- "/dev ": 시스템 디바이스 파일들을 저장하는 디렉토리, 하드디스크 장치파일, CD-ROM 장치 파일 같은 파일 저장. 

  - 하드디스크, 프리넡, 입출력장치 등과 같은 장치들을 파일화하여 관리한다. -> "리눅스의 모든 것은 파일이다"

  - 따라서 특정 장치를 실행하기 위해서는 해당 장치 파일을 실행해야 한다. -> 사용자가 마우스를 움직이거나 하드디스크에 파일을 저장할 때, 운영체제(커널)는 내부적으로 /dev에 있는 장치 파일을 매개체로 삼아 하드웨어와 끊임없이 상호작용한다.

  - 해당경로 접근 예시
    - 우리가 USB에 파일을 복사한다고 가정해보자
    - 애플리케이션 계층 : 복사 프로그램이 USB데이터를 써줘라는 요청을 함
    - 커널 계층 : 운영체제는 USB와 연결된 장치 파일(예 : /dev/sdb)을 찾아 그 파일에 데이터 쓰기 작업을 수행한다.
    - 드라이버 및 하드웨어 : 커널 내의 "장치 드라이버"가 해당 파이렝 써진 데이터를 실제 전기 신호로 바꿔 USB메모리 칩에 기록

  - 해당 경로에 있는 파일은 
     - Device File -> 장치 드라이버
     - Block Device -> 하드디스크등의 저장장치
     - Character Device-> 입출력장치들

  - 리눅스의 표준 입력 장치는 키보드, 표준 출력 장치는 모니터이다. 

- "/etc" : 시스템 환경 설정 파일 저장 디렉토리
  - 시스템 환경설정 파일과 부팅 관련 스크립트 파일들이 저장 되어있는 디렉토리
  - 사용자 정보 및 암호 정보 파일, 보안 파일등을 저장 
  
- "/root" : 시스템 고나리자용 홈 디렉터리
  
- "/sbin" " 관리자용 시스템 표준 명령 및 시스템 관리와 관련된 실행 명령어 저장 -> usr/sbin의 심볼링크이다. 
  
- "/usr" : 사용자 디렉토리로 사용자 데이터가 애플리케이션 저장
  - 시스템이 아닌 일반 사용자들이 사용하는 디렉토리
  - 공유 가능한 프로그램들이 설치되며 네트워크를 이용해서 여러 개의 시스템을 연결할 경우 이 디렉토리를 공유해서 설치된 프로그램들을 활용할 할 수 있다. -> "똑같은 프로그램을 여러 컴퓨터에 일일이 설치하지 말고, 한 곳에만 설치한 뒤 네트워크로 연결해서 다 같이 쓰자"
  - /usr 디렉토리는 읽기 전용으로 마운트 되어야하며, 가변 자료들은 /var 디렉토리로 심볼링 링크로 사용하게 된다. -> 그래야 해커들이 해당 프로그램을 바꾸는 등 보안적으로 안전하다. 필요할 때만 읽기/쓰기로 재마운트 하여 쓰고, 다시 읽기전용으로 마운트해야한다. 
 
  
- "/home" : 사용자 계정 디렉토리로 여러 계정들의 홈 디렉터리가 위치함, 일반 사용자들이 로그인 시 처음으로 위치하게 되는 디렉토리임
  
- "/var" : 가변 자료 저장 디렉토리 로그 파일이나 메일 데이터 저장
  - 시스템에서 사용된느 가변적인 파일들을 저장하는 디렉토리
  - 가변적인 파일들로는 로그파일, 스풀링, 캐싱 등이 있다. 
  
- "/tmp" : 각종 프로그램이나 프로세스 작업을 할 때 임시로 생성되는 파일 저장, 모든 사용자에 대해서 읽기와 쓰기가 허용, 스티키 비트 설정으로 파일의 소유자만이 자신의 소유 파일 삭제 가능.
  
- "/mnt" : 파일 시스템을 일시적으로 마운트할 때 사용
  
- "/lost+found" : 결함이 있는 파일에 대한 정보가 저장되는 디렉토리
  - 파일 시스템의 이상 유무를 진단하고 복구하는 "fsck"(file system check)에 의해서 사용되는 디렉토리
  - 손상된 파일이나 디렉토리를 이 디렉토리로 연결한 뒤에 오류를 수정하게 된다
  - 평상 시에는 null파일 리읔에 의해서 비어있는 상태로 존재한다.
  - 리눅스 파일 시스템 ext2에 의한 fsck.ext2프로그램도 이 디렉토리를 사용한다. 

+ (추가)부팅시 시나리오
- BIOS/UEFI가 /boot의 부트로더를 실행.

- 부트로더가 커널을 로드하고, 커널은 /lib의 모듈을 로드하여 하드웨어를 제어.

- 기본 시스템이 준비되면 /usr를 마운트하여 쉘(Shell)과 GUI 환경을 제공.

- 사용자가 로그인하면 /home/user의 개인 설정 파일(.bashrc 등)을 읽어 사용자 환경을 완성.


# Section2 . 리눅스의 역사

## [UNIX의 탄생과 발전]

- 1965년, MIT, AT&T 벨 연구소, 그리고 General Electyric은 시분할 운영체제 multics의 공동개발 프로젝트를 시작
- 그러나 개발이 지연되고 복잡해지면서 프로젝트가 철수됐으며, 이후 켄톰슨과 데이나 리치가 multic의 개념의 단순화한 UNIX를 개발하였다.
- 1972년 벨 연구소의 데니스 리치가 C언어를 개발하였으며 1973년 켄 톰슨과 함께 UNIX커널을 C로 재작성하였다.
- 이로써 UNIX는 이식성과 호한성이 뛰어난 시스템으로 발전하였으며, 소스 코드가 대학과 연구기관에 배포되면서 버클리대의 BSD Unix와 벨 연구소의 system V UNIX로 각각 발전하게 되었다.

## [자유 소프트웨어 운동의 등장]

- MIT 인공지능 연구소의 연구원이었던 리처드 스톨만은 소스 코드를 비공개로 전환하고 소프트웨어를 상업화하려는 흐름에 반발하여, 1983년 자유 소프트웨어 운동의 일환으로  GNU프로젝트를 시작했다.
- 리처드 스톨만은 1983년 GNU프로젝트를 시작하고 1985년에 GNU 선언문을 발표했다.
- 또한 그는 자유 소프트웨어 운동을 제도적으로 지원하기 위해 비영리단체인 FSF(Free software foundation)을 설립했다.
- 1987년 앤드류 s 타넨바움은 운영체제 교육용 운영체제인 미닉스(MINIX)를 개발하고 공개했다
- 미닉스는 처음에는 교육용으로만 무료로 공개되었지만 이후에 오픈소스 라이선스로 전환됐다.

## [리눅스 커널의 탄생과 성장]

- GNU 프로젝트는 1980년대 후반 자체 커널인 Hurd를 개발 중이었으나 완성되지 못했다.
- 1991년 리누스 토르발즈가 개발한 리누스 커널이 1992년 GPL 라이선스를 채택하면서 GNU의 도구와 결합되면서 GNU/Linux 시스템이 탄생했다.

# Section3 . 리눅스 라이선스

## [자유 소프트웨어 운동과 GNU 프로젝트]

[1] GNU(GNU’s Not Unix)

- 리처드 스톨만이 자유 소프트웨어 재단에서 진행하여 유지중인 운영체제 프로젝트
- GNU 프로젝트를 통하여 개발한 유닉스 계열 컴퓨터 운영체제로 “완전한 유닉스 호환 소프트웨어 시스템” 이 되는 것이 목표이다.

[2] 자유 소프트웨어 재단 FSF

- 1985년 리처드 스톨만이 설립한 재단이다.
- 자유 소프트웨어는 사용자가 소프트웨어를 실행하고, 복제하고, 배포하고, 학습하고, 개작하고, 향상시킬 수 있는 소프트웨어이다.
- 자유 소프트웨어의 특징
    - 어떤 목적이든 원하는 대로 프로그램을 실행시킬 수 있는 자유
    - 무료 또는 유료로 프로그램 복제물을 재배포할 수 있는 자유
    - 필요에 따라 프로그램을 개작할 수 있는 자유
    - 공동체 전체가 개선된 이익을 나눌 수 있게 개작한 프로그램을 배포할 수 있는 자유
- 자유는 금전적인 측면에서 신경을 안 쓰기 때문에 자유 소프트웨어를 유료로 판매하는게 문제가 되지는 않는다.

[3] 오픈 소스 소프트웨어

- 1998년 일부 커뮤니티에서 자유 소프트웨어 대신 오픈소스 소프트웨어 라는 용어를 사용하기 시작했다.
- 이것은 자유가 가진 무료라는 의미가 일으키는 혼동을 피하기 위함.
